# 智能周报 Agent 验收测试文档

> 本文档用于测试 Markdown Agent 的结构化编辑能力。
> 重点测试能力包括：读取 section、替换 section、在指定标题后插入内容、识别代码块、更新任务列表、格式化 Markdown，并避免误改受保护章节。

## 1. 背景

团队每周需要从项目记录、任务列表、风险登记表和会议纪要中整理项目周报。当前周报主要由项目负责人手工汇总，存在遗漏信息、格式不统一、风险描述不完整等问题。

### 1.1 当前问题

* 多个信息来源分散，整理成本高。
* 不同成员填写习惯不一致，导致周报风格不统一。
* 风险项经常只写现象，没有说明影响和缓解方式。
* 历史周报难以按项目、负责人、风险类型进行检索。

### 1.2 目标

* 自动汇总项目进展、风险和待办事项。
* 支持按 Markdown section 精准读取和修改内容。
* 保持周报结构一致，便于人工审阅和后续归档。
* 支持人工只修改局部章节，而不是重写整份文档。

### 1.3 非目标

* 不生成绩效评价。
* 不替代项目负责人的最终确认。
* 不做复杂视觉排版。
* 不接入即时通信工具的实时消息流。

## 2. 用户场景

### 场景一：生成周报草稿

用户上传项目记录后，Agent 根据固定模板生成一份周报草稿，包括本周进展、风险、待办事项和下周计划。

### 场景二：局部更新风险

用户发现风险描述不清晰，只希望 Agent 修改“风险”章节，不希望影响背景、用户场景、FAQ 等稳定内容。

### 场景三：插入补充说明

用户希望在指定章节后插入新的小节，例如在“数据来源”章节中补充“数据更新频率”。

### 场景四：勾选任务状态

用户希望 Agent 根据修改结果同步更新待办清单，例如将“支持替换风险章节”从未完成改为已完成。

## 3. 功能需求

### 3.1 读取指定章节

Agent 应能根据完整标题路径读取指定 section，而不是只根据标题文本进行模糊匹配。

示例请求：

```json
{
  "heading_path": ["3. 功能需求", "3.1 读取指定章节"]
}
```

读取时需要注意：

* 标题路径必须精确匹配。
* 不应把代码块中的 Markdown 标题误识别为真实标题。
* 遇到重名标题时，应优先使用完整标题路径消除歧义。

### 3.2 替换指定章节

Agent 应能替换某个 section 的正文，并保留该 section 的标题。

适合测试的章节包括：

* `## 5. 风险`
* `### 3.2 替换指定章节`
* `## 8. 验收标准`

替换时需要注意：

* 不要误删后续同级章节。
* 不要修改受保护章节。
* 如果 section 内部包含子标题，应明确替换范围是否包含子标题。

### 3.3 插入新章节

Agent 应能在指定 heading 后插入新的小节或正文内容。

本节是插入测试点。测试时可以要求 Agent 在本节正文之后、`### 3.4 更新代码块` 之前插入一个新的四级标题。

示例插入内容：

```markdown
#### 示例插入标题

这里是示例插入正文。
```

### 3.4 更新代码块

Agent 应能识别文档中的代码块，并避免把代码块中的标题样式文本当作真实 Markdown 标题。

```python
def generate_weekly_report(project_name: str, include_risks: bool = True) -> str:
    """Generate a weekly report title."""
    if include_risks:
        return f"# {project_name} 周报\n\n## 5. 风险\n\n- 待补充风险"
    return f"# {project_name} 周报"
```

说明：

* 上方代码块中包含 `## 5. 风险` 字符串。
* Agent 不应把该字符串识别为真实章节。
* 除非明确要求，否则不要修改代码块内容。

### 3.5 勾选任务

Agent 应支持更新 Markdown 任务列表状态。

可测试的任务包括：

* 将未完成任务改为已完成。
* 保留无关任务状态不变。
* 保留嵌套任务的层级结构。

## 4. 数据来源

周报 Agent 需要从多个来源读取信息，并根据来源类型构建结构化输入。

| 来源    | 说明              | 是否必需 | 责任人   |
| ----- | --------------- | ---- | ----- |
| 项目记录  | 每日进展、变更说明、阶段性成果 | 是    | 项目负责人 |
| 任务列表  | 当前待办、负责人、完成状态   | 是    | 项目经理  |
| 风险登记表 | 风险、阻塞项、影响范围、负责人 | 否    | 项目负责人 |
| 会议纪要  | 关键决策、遗留问题、行动项   | 否    | 会议记录人 |

### 4.1 来源校验

在生成周报前，需要对数据来源进行基础校验：

* 检查必需来源是否存在。
* 检查任务是否包含负责人和状态。
* 检查风险项是否包含影响说明。
* 检查会议纪要是否存在明确行动项。

## 5. 风险

### 目标

这里的“目标”标题用于测试重名标题处理。注意，文档中已经存在 `### 1.2 目标`，Agent 不应仅凭“目标”二字进行错误匹配。

* 输入信息不完整，可能导致周报遗漏关键进展。
* 自动总结可能过度压缩，导致重要上下文丢失。
* 任务状态可能与真实项目状态不同步。
* 风险登记表可能缺少负责人，导致后续无法追踪。
* 多来源内容可能存在冲突，导致周报结论不一致。

## 7. FAQ

### Q1：为什么要按 section 编辑？

因为按 section 编辑可以减少误改无关内容的风险，也便于在长文档中进行局部修改。

### Q2：标题重名怎么办？

演示版本中可以要求用户提供完整标题路径，例如 `["1. 背景", "1.2 目标"]`。

### Q3：为什么不能直接全文重写？

全文重写容易破坏人工已经确认过的内容，也容易导致格式、编号和引用关系发生不必要变化。

## 8. 验收标准

本测试文档的验收重点包括：

| 验收项   | 预期结果                      |
| ----- | ------------------------- |
| 读取章节  | 能准确读取指定标题路径下的内容           |
| 替换章节  | 能替换 `## 5. 风险`，且不影响其他章节   |
| 插入章节  | 能在指定标题后插入新小节              |
| 保护章节  | 不修改背景、用户场景和 FAQ           |
| 代码块识别 | 不把代码块中的 `## 5. 风险` 当作真实标题 |
| 格式化   | 表格、列表、空行和标题格式保持规范         |
