### 1.1 Agent 是什么？（用一句话说清楚）

你最近可能听说过"小龙虾"（Manus）——那个能自动做研究、写报告、分析数据的 AI。它之所以这么火，是因为它不只是"聊天"，而是能**真正做事**。

如果你用过 ChatGPT 或 Claude，你已经接触过大语言模型（LLM）。你问它问题，它回答你。这就像有一个知识渊博的朋友，你问什么它答什么——但它**只能动嘴，不能动手**。

**Agent = 大语言模型 + 工具 + 自主行动的能力**

小龙虾就是 Agent 的一个经典代表。它能在浏览器里搜索资料、运行代码分析数据、生成图表和报告——像一个全能助手。

打个比方：

| 概念 | 比喻 | 特点 |
|------|------|------|
| **纯 LLM**（ChatGPT） | 一个坐在书桌前的学者 | 知识渊博，但只会说不会做 |
| **Agent**（小龙虾/我们的 Agent） | 一个走进实验室的科学家 | 能思考、能查资料、能动手操作、能根据结果调整策略 |

Agent 的核心循环只有四步：

```
看到（感知）→ 思考（决策）→ 行动（执行）→ 看到结果（反馈）
                              ↕
                        继续循环，直到完成
```

这个循环我们会在整门课中反复提到。

## 第2章：Agent 的大脑 — 大语言模型在思考什么

### 本章目标

- 理解大语言模型（LLM）的工作原理（通俗版）
- 知道 API 调用是怎么回事
- 看懂 Agent 是如何"连接"大模型的
- 理解模型参数的含义

---

### 2.1 大语言模型不是一个"数据库"

很多人以为大语言模型（LLM）就像一个巨大的数据库——你问一个问题，它去数据库里"查"答案。

**不对。**

大语言模型更像一个**超级预测引擎**：

> 给它一段话的前半部分，它预测最可能的下一个字/词是什么。

举例：
```
用户输入："中国的首都是____"
模型预测：最可能是"北京"
```

```
用户输入："苹果是一种____"
模型预测：最可能是"水果"（而不是"公司"）
```

你看到的每一段流畅的回答，其实都是模型**一个字一个字地预测**出来的。每一次预测都考虑了前面所有已经生成的内容。

> 💡 **这为什么重要？**
> 因为这个机制决定了 Agent 的工作方式。Agent 不是在"检索答案"，而是在"生成答案"。它会把工具返回的信息当作新的"上下文"塞进预测过程，从而生成基于真实数据的回答。

---

### 2.2 从 LLM 到 Agent：多了一层"思考层"

纯 LLM 的工作方式：

```
你：今天天气怎么样？
LLM：抱歉，我无法获取实时天气信息。
```

它知道自己的局限——它没有获取实时信息的能力。

Agent 的工作方式：

```
你：今天天气怎么样？
Agent 内部思考：
  → 用户想知道天气 → 我需要查天气 → 我有一个 get_weather 工具
  → 调用工具获取数据 → 拿到结果 → 基于结果生成回答
你：今天北京天气晴，15-25°C...
```

关键区别：**Agent 会"思考"该用什么工具，然后主动去获取信息。**

---

### 2.3 Agent 是如何"连接"大脑的？

在代码层面，Agent 连接大模型只需要几行关键配置。来看一个简化版本：

```python
# 创建模型连接（关键代码）
model = DeepSeekChatModel(
    credential=DeepSeekCredential(api_key="sk-xxx"),
    model="deepseek-chat",
)
```

这三行在做什么？

| 部分 | 比喻 | 作用 |
|------|------|------|
| `api_key` | 你的身份证 | 证明你有权限使用这个模型 |
| `model="deepseek-chat"` | 你找哪位专家 | 指定使用哪个模型版本 |
| `DeepSeekChatModel` | 电话线 | 建立和模型服务的连接 |

**代码不重要，理解这个连接方式才重要**：Agent 通过 API（应用程序接口）和大模型"通话"——把你的问题发送过去，接收模型生成的结果。

整个过程是：

```
你的输入 → Agent 包装成消息格式 → 发送到模型 API → 模型返回文字 → Agent 解析结果
```

---

### 2.4 模型的"性格"可以调节

你有没有想过，为什么有时候模型回答很严谨，有时候又很有创意？

模型有几个关键参数可以调节它的"性格"：

| 参数 | 通俗理解 | 取值范围 | 在我们的项目中 |
|------|---------|---------|---------------|
| **temperature**（温度） | 创意的程度 | 0 ~ 2 | 默认 1.0 |
| **max_tokens**（最大长度） | 最多说多少字 | 视模型而定 | 可配置 |

**temperature 的作用：**

- **temperature = 0**：每次回答几乎一样，像照本宣科的播音员
- **temperature = 0.7**：有一些变化，像正常对话
- **temperature = 1.5**：非常随机，像喝醉了说话

在我们的项目中，这些参数是通过配置文件来管理的（而不是写死在代码里），这样修改起来很方便：

```toml
# config/agentscope.toml
[model]
model = "deepseek-chat"
temperature = 0.7
max_tokens = 2048
```

> 💡 **为什么用配置文件而不是写死在代码里？**
> 因为参数需要反复调试。写在配置文件中，改参数不用改代码，更灵活。这也是实际项目中的最佳实践。

### 2.5 AgentScope 帮我们做了什么？

你可能注意到了，我们的代码没有直接处理"怎么调用 API"、"怎么处理网络错误"、"怎么解析返回结果"这些麻烦事。

这是因为 AgentScope 帮我们封装好了：

```
你的代码
    ↓
AgentScope 框架  ←—— 帮你处理了：
  ├─ API 调用细节       （不用写网络请求代码）
  ├─ 消息格式转换        （不用拼 JSON）
  ├─ 流式输出处理        （不用处理数据流）
  ├─ 错误重试           （不用写 try-catch）
  └─ 工具调用管理        （不用自己调度）
    ↓
大模型 API
```

**这就是框架的价值**：让你专注于"我要做什么"，而不是"我怎么连接网络"。

## 第3章：Agent vs 纯 LLM vs 工作流 — 谁更强？

### 本章目标

- 理解三种方案的各自特点
- 知道什么时候该用 Agent，什么时候不需要
- 建立选择框架：什么场景用 Agent 最合适

---

### 3.1 三种方案，一个比喻

想象你要做一顿饭：

| 方案 | 比喻 | 特点 |
|------|------|------|
| **纯 LLM** | 一本食谱 | 知识丰富，告诉你每一步怎么做，但它自己不动手 |
| **工作流** | 自动炒菜机 | 按固定程序运行，每步都预设好了，遇到意外就卡住 |
| **Agent** | 一个大厨 | 能看、能想、能动手，发现缺盐会自己去找，火候不对会调整 |

---

### 3.2 对比一：纯 LLM

纯 LLM 就是你现在使用的 ChatGPT、Claude 等聊天工具。

**工作方式：**
```
用户输入 → LLM → 文字输出
```

**优势：**
- 使用简单，开箱即用
- 知识面广
- 适合问答、翻译、写作等纯文本任务

**局限：**
- ❌ 不能获取实时信息（知识截止到训练时）
- ❌ 不能操作你的文件
- ❌ 不能执行实际动作
- ❌ 每次对话独立，没有"记忆"之外的上下文

**适合场景：** 写作辅助、翻译、代码解释、头脑风暴

---

### 3.3 对比二：工作流

工作流是预定义的、固定的步骤序列。比如你要处理一个 Markdown 文件：

```python
# 工作流：固定步骤，不可变通
def process_markdown(file_path):
    # 步骤1：读取文件（一定会执行）
    content = read_file(file_path)
    # 步骤2：提取大纲（一定会执行）
    outline = extract_outline(content)
    # 步骤3：格式化（一定会执行）
    formatted = format_markdown(content)
    # 步骤4：保存（一定会执行）
    save_file(file_path, formatted)
    return outline
```

**工作方式：**
```
输入 → 步骤1（固定）→ 步骤2（固定）→ 步骤3（固定）→ 输出
```

**优势：**
- 稳定可靠，每次执行结果一致
- 容易调试，哪一步出问题一目了然
- 执行速度快，没有"思考"开销

**局限：**
- ❌ 缺乏灵活性——遇到预设外的场景就失效
- ❌ 不会根据中间结果调整后续步骤
- ❌ 不能处理"如果……就……否则……"的复杂逻辑（除非预判所有分支）
- ❌ 维护成本高——每新增一个场景就要改代码

**适合场景：** 数据清洗、定时任务、格式转换、CI/CD 流水线

---

### 3.4 对比三：Agent

Agent 结合了前两者的优点——有 LLM 的理解能力，又有执行动作的能力，还能自主决策。

**先看一个你肯定听过的例子：小龙虾（Manus）**

你刷新闻的时候可能看过"小龙虾自动写研究报告"的演示。它是怎么做到的？

```
你：帮我研究一下2026年新能源汽车市场趋势

小龙虾（Agent）：
  → 第1步：打开浏览器，搜索最新资料
  → 第2步：读取搜索结果中的文章
  → 第3步：分析关键数据和趋势
  → 第4步：生成图表（调用了代码工具）
  → 第5步：整理成一份完整的报告
  → 给你：一份基于真实数据、带图表的研究报告
```

整个过程没有人工干预。小龙虾自己决定先搜索、再分析、再生成图表、最后排版。这就是 Agent 的威力——**它自己规划步骤，自己调用工具，自己判断结果**。

**那小龙虾和我们的 Agent 有什么关系？**

答案是——**原理一模一样**。

小龙虾有几十个工具（浏览器、代码执行器、文件读写等）。我们的 Markdown Agent 只有 9 个工具（扫描、读取、编辑、格式化）。但核心机制完全相同：

```
小龙虾                      我们的 Agent
─────                       ───────────
LLM 大脑                    DeepSeek 模型
浏览器工具                  扫描/读取工具
代码执行器                  编辑/格式化工具
自主决策循环                 自主决策循环
```

你把我们的 Agent 加上浏览器工具，它也能上网搜索。加上代码执行器，它也能运行代码。**原理相同，只是工具数量不同。**

**回到对比本身：**

**工作方式：**
```
用户输入
    ↓
Agent"思考"：需要做什么？
    ↓
选择工具 → 执行 → 观察结果
    ↓
根据结果决定下一步
    ↓
（循环直到任务完成）
    ↓
输出最终答案
```

**优势：**
- ✅ 能理解复杂、模糊的指令
- ✅ 能自主决定调用什么工具、按什么顺序
- ✅ 能根据中间结果调整策略
- ✅ 可以做 LLM 做不到的事（操作文件、查询数据）

**局限：**
- ⚠️ 执行速度较慢（思考需要时间）
- ⚠️ 可能做出错误决策（需要约束机制）
- ⚠️ 对提示词质量敏感

**适合场景：** 代码开发、数据分析、文档处理、自动化运维

---

### 3.5 直观对比

用一个真实任务来对比三种方案：**"把 Test.md 中第 5 个任务标记为已完成"**

| 方案 | 怎么做 | 代码量 | 灵活性 |
|------|--------|-------|--------|
| **纯 LLM** | 告诉它"把任务5标记为完成"——但它做不到，它只能告诉你"你应该找到第5行把[ ]改成[x]" | 0（但做不了） | ❌ |
| **工作流** | 写死步骤：读取文件→解析任务→找到第5个→替换→保存。如果文件结构变了就失效 | ~30行 | ❌ |
| **Agent** | 告诉它"把任务5标记为完成"——它自己决定调用 markdown_list_tasks 找到任务，再调用 markdown_update_task_status 修改 | 0（框架处理） | ✅ |

```python
# ── 工作流方式：必须精确指定每一步 ──
def mark_task_done(filepath, task_index):
    lines = read_lines(filepath)           # 固定步骤1
    task_line = find_task_line(lines, task_index)  # 固定步骤2
    lines[task_line] = lines[task_line].replace("[ ]", "[x]")  # 固定步骤3
    write_lines(filepath, lines)           # 固定步骤4

# ── Agent 方式：只需告诉它目标 ──
# 用户说："把 Test.md 的第5个任务标记为完成"
# Agent 自己决定调用什么工具、按什么顺序
```

### 3.6 三种方案的选择框架

不是所有场景都适合用 Agent。一张表帮你做选择：

```
你的任务需要？
    │
    ├─ 纯文本生成（写作、翻译、问答）
    │   └─ → 纯 LLM ✓
    │
    ├─ 固定流程、重复操作（定时任务、数据清洗）
    │   └─ → 工作流 ✓
    │
    ├─ 需要理解上下文 + 执行操作（开发、文档处理）
    │   └─ → Agent ✓
    │
    └─ 三者结合
        └─ → Agent 调用工作流 ✓
```

> 💡 **实际项目中三者在共存**
> 最强大的方案是三者结合：
> - 用**工作流**处理确定性的重复任务
> - 用**Agent**做决策和调度
> - Agent 在工作过程中调用**LLM**进行理解、总结、生成

### 3.7 从代码看本质区别

来看我们项目中 Agent 的核心循环（简化版）：

```python
# Agent 的核心：循环——感知→思考→行动→反馈
while 任务未完成:
    思考当前情况                      # LLM 理解上下文
    决定下一步做什么                   # LLM 做决策
    如果需要工具:
        执行工具调用                   # 动手做事
        把结果加入"记忆"               # 更新上下文
    继续思考                         # LLM 评估结果
```

这个循环就是 Agent 和另外两种方案的本质区别：

- **纯 LLM**：只有"思考"（生成文字），没有"行动"
- **工作流**：只有"行动"（固定步骤），没有"思考"
- **Agent**：既有"思考"又有"行动"，还能根据反馈调整

## 第4章：Agent 的双手 — 工具调用是如何工作的

### 本章目标

- 理解什么是"工具"以及为什么 Agent 需要工具
- 看懂工具的定义方式（函数 + 描述 = 工具）
- 理解只读工具和写工具的区别
- 了解权限控制机制

---

### 4.1 为什么 Agent 需要工具？

想象你是一个知识渊博的人，被关在一个没有窗户的房间里。别人问你：

- "外面的天气怎么样？"——你不知道，你看不到窗外
- "帮我查一下这个文件的内容"——你碰不到文件
- "帮我把这个表格的第三行改一下"——你改不了

纯 LLM 就是这个处境——它有很多知识，但它**感知不到你的电脑，也操作不了你的文件**。

工具就是 Agent 的"感觉器官"和"手脚"：

```
工具类型        比喻            作用
─────────      ──────          ──────────
扫描工具    →  眼睛          → 看到文件在哪里
读取工具    →  手指          → 翻开文件看内容
编辑工具    →  笔            → 修改文件内容
检查工具    →  校对员        → 检查格式是否正确
```

---

### 4.2 工具的本质：一个函数 + 一段描述

在代码层面，工具本质上就是一个**普通的 Python 函数**，加上一段描述性的"说明书"。

来看一个简化版本：

```python
# 这是一个"工具"——本质上就是一个函数 + 描述
def scan_files(path=None):
    """扫描目录中的 Markdown 文件"""
    # ... 实际的扫描逻辑 ...
    return ["README.md", "Test.md", "docs/guide.md"]
```

要让 Agent 能使用这个函数，需要给它贴上"标签"：

```python
# 把普通函数注册为 Agent 可用的工具
scan_tool = FunctionTool(
    scan_files,
    is_read_only=True,    # 这个工具只读，不会修改文件
)
```

**关键理解**：工具不需要复杂的实现。它就是你已经会写的 Python 函数，加一个"注册"步骤，Agent 就能用了。

---

### 4.3 Agent 如何"学会"使用工具？

Agent 不是预先编程知道每个工具的用法。它的工作方式是：

**步骤1：告诉 Agent 有哪些工具可用**
在 Agent 启动时，我们给它一份"工具清单"：

```
你当前可用的工具：
1. scan_directory —— 扫描目录中的文件
2. outline —— 提取文件大纲
3. get_section —— 读取文件的某个章节
4. replace_section —— 替换文件的某个章节
5. update_task —— 更新任务状态
```

**步骤2：Agent 根据任务选择工具**
当你提出请求，Agent 会"思考"该用哪个工具：

```
用户说："帮我看看 Test.md 的结构"
Agent 思考：
  → 用户想看文件结构
  → 我有一个 outline 工具可以提取大纲
  → 好的，调用 outline(path="Test.md")
```

**步骤3：Agent 读取工具结果并生成回答**
工具返回数据后，Agent 把数据当作"已知信息"，基于它生成回答。

---

### 4.4 只读 vs 写操作：一个重要的设计

在我们的项目中，工具被分为两类：

| 类型 | 例子 | 比喻 | 权限 |
|------|------|------|------|
| **只读工具** | 扫描文件、读取内容、检查格式 | 只看不动 | ✅ 自动允许 |
| **写工具** | 替换章节、插入内容、修改任务 | 动手修改 | ⚠️ 需要确认 |

这个设计背后的逻辑很实用：

```python
# 只读工具 —— 安全，自动放行
read_tool = FunctionTool(scan_files, is_read_only=True)
# Agent 可以随时调用，不需要问你

# 写工具 —— 有风险，需要你确认
write_tool = FunctionTool(replace_section, is_read_only=False)
# Agent 想调用时，会先弹出确认：
# ⚠️ Tool permission required: replace_section
#    args: {"path": "Test.md", "heading": "风险"}
#    Allow this tool call? [y/N]:
```

**为什么需要确认？**
因为 Agent 可能做出你不期望的修改。在实际使用中，你希望 Agent 能自由地"看"任何东西，但"改"任何东西之前都应该问一下你。

## 第5章：看得见的思考 — Codex 风格交互体验

### 本章目标

- 知道为什么实时展示 Agent 的"思考过程"很重要

---

### 5.1 为什么"看得见思考"很重要？

想象两个场景：

**场景 A：黑盒**
```
你：帮我分析一下这个文件
（等待 5 秒）
🤖：分析完成，文件有 3 个问题。
```
你不知道 Agent 在这 5 秒里做了什么。是卡住了？在思考？在调用工具？失败了重试？

**场景 B：透明**
```
你：帮我分析一下这个文件
🤖 Assistant: 好的，我先读取文件内容...
🔧 Tool: get_section
   args: {"path": "Test.md"}
   output: （文件内容摘要）
   result: success
🤖 Assistant: 我看到文件包含 8 个章节...
🔧 Tool: list_tasks
   args: {"path": "Test.md"}
   output: 共 8 个任务，3 个已完成
   result: success
🤖 Assistant: 分析完成，文件有 3 个问题...
```

**透明模式的好处：**
1. ✅ 你知道 Agent 没有"偷懒"——它在真实地工作
2. ✅ 如果出错了，你能看到哪一步出了问题
3. ✅ 更信任 Agent——因为你看得到它的工作过程
4. ✅ 更高效——你可以提前中断或纠正

这就是 **Codex / Claude Code 风格交互** 的核心设计理念：**让 Agent 的思考过程可见**。

---

### 5.2 流式输出：文字"流"出来

你有没有注意到，ChatGPT 的回答是一个字一个字"冒"出来的？这就是**流式输出（Streaming）**。

**传统方式（非流式）：**
```
发送请求 → 等待 5 秒 → 一次性收到完整回答
你在这 5 秒内只能盯着"正在输入…"的提示
```

**流式方式：**
```
发送请求 → 马上开始收到文字 → 文字不断"流"出来
你几乎感觉不到等待——文字实时出现
```

在我们的 Agent 中，流式输出的渲染是这样处理的（简化概念）：

```
模型生成："我" → 显示 "我"
模型生成："是" → 显示 "是"  
模型生成："你" → 显示 "你"
模型生成："的" → 显示 "的"
模型生成："助" → 显示 "助"
模型生成："手" → 显示 "手"
```

每个字一生成就显示，而不是等全部生成完再显示。

---

### 5.3 工具调用的实时展示

在 Agent 调用工具时，我们不仅展示工具的结果，还展示**调用的全过程**：

```
🔧 Tool: markdown_get_section          ← 显示调用了什么工具
   args: {"path": "Test.md", ...}       ← 显示传入的参数
   output:                               ← 显示工具返回的结果
      ## 5. 风险
      - 风险一：...
   result: success                       ← 显示执行状态
```

这背后的处理逻辑（简化概念）：

```
当 Agent 开始调用工具时：
  → 显示 "🔧 Tool: 工具名"
  → 显示参数（实时流式传入）
  → 工具执行中显示 "status: running"
  → 工具返回后显示结果预览
  → 显示 "result: success/fail"
当 Agent 继续生成回答时：
  → 显示 "🤖 Assistant: "
  → 流式显示文字
```

---

### 5.4 终端交互 vs 网页聊天

我们的 Agent 运行在**终端（命令行）**中，这和你在网页上使用 ChatGPT 有很大不同：

| 维度 | 网页聊天 | 终端交互 |
|------|---------|---------|
| **启动方式** | 打开浏览器 | 敲一行命令 |
| **界面** | 图形界面、按钮、图标 | 纯文字 |
| **资源占用** | 浏览器可能占几百 MB 内存 | 终端几乎不占资源 |
| **集成度** | 在网页里独立运行 | 和你的文件系统在一起 |
| **自动化** | 难 | 容易（可以嵌入脚本） |
| **适用场景** | 日常对话、写作 | 开发、文件操作、自动化 |

**为什么我们的 Agent 选择终端？**
因为我们的 Agent 要操作你的文件——它和你的项目在同一个目录下，可以直接读写文件。这在网页里是做不到的。

---

### 5.5 "透明 Agent"的设计原则

从我们的项目可以总结出几个设计原则，适用于任何 Agent 交互：

**原则1：不要隐藏 Agent 的思考过程**
```
✅ 好：显示工具调用、参数、结果
❌ 不好：只显示最终答案
```

**原则2：实时反馈优于批量反馈**
```
✅ 好：一个字一个字地显示
❌ 不好：等全部生成完再显示
```

**原则3：区分"思考"和"输出"**
```
✅ 好：工具调用用 🔧，回答用 🤖，一目了然
❌ 不好：所有东西混在一起
```

**原则4：给用户控制权**
```
✅ 好：写操作需要确认
❌ 不好：Agent 直接修改文件
```

### 6.2 场景一：快速了解文档

### 6.6 Agent 的"超能力"来自哪里？

回顾整个工作过程，Agent 的"超能力"来自三个关键设计：

**1. 工具的组合使用**

Agent 不是只会用一个工具。它能把多个工具组合起来完成复杂任务：

```
读取大纲（了解结构）
    ↓
列出任务（查看状态）
    ↓
更新任务（修改内容）    ← 每个步骤之间，Agent 都在"思考"
    ↓                    ← 根据上一步结果决定下一步
插入新内容（补充信息）
    ↓
格式化文件（收尾）
```

**2. 自然语言交互**

你不需要记住任何命令或 API。用日常语言就能指挥 Agent：

```
❌ 不用写：markdown_update_task_status("Test.md", 4, true)
✅ 只需说："帮我把读取指定章节那个任务标记为完成"
```

**3. 透明的过程**

每一步都展示给你看，你可以随时确认或纠正：

```
🔧 调用工具 → 展示参数 → 请求确认 → 展示结果 → 继续下一步
```

### 6.7 Agent 做不到什么？（坦诚面对局限）

任何技术都有边界。我们的 Agent 目前：

| 能做的 | 还不能做的 |
|--------|-----------|
| 读取和编辑 Markdown 文件 | 处理 Word/Excel/PDF |
| 扫描目录中的文件 | 访问互联网搜索 |
| 更新任务状态 | 运行 Python 代码 |
| 格式化文档 | 与外部 API 交互 |
| 展示工具调用过程 | 多 Agent 协作 |

这些"还不能做的"就是未来的扩展方向。

但有意思的是：小龙虾能做的事，和我们的 Agent 能做的事，本质区别只是**工具数量**。小龙虾有几十个工具，我们只有 9 个。但核心机制——"LLM + 工具 + 自主决策"——完全一样。你想让你的 Agent 也能上网搜索？给它加一个浏览器工具就行。这就是从 0 到 1 搭建 Agent 的意义：你理解了核心原理，剩下的只是"加工具"。

### 7.1 回顾：Agent 的完整工作链路

经过六章的学习，我们可以画出 Agent 的完整工作链路了：

```
你输入问题
    │
    ▼
┌─────────────────────────────────────┐
│            Agent 循环                │
│                                      │
│  ① 感知：接收你的输入                │
│     ↓                               │
│  ② 思考：LLM 理解并决定做什么        │
│     ↓                               │
│  ③ 决策：需要调用工具吗？            │
│     ├── 不需要 → 直接生成回答         │
│     └── 需要 →                        │
│          ├─ 选择工具                  │
│          ├─ 填入参数                  │
│          ├─ 需要权限？→ 问你          │
│          ├─ 执行工具                  │
│          ├─ 获取结果                  │
│          └─ 回到 ②（继续思考）        │
│     ↓                               │
│  ④ 输出：生成最终回答                │
│                                      │
└─────────────────────────────────────┘
    │
    ▼
你在终端看到回答
```

这个循环中的每一步，都有相应的代码组件负责：

| 步骤 | 组件 | 作用 |
|------|------|------|
| ① 感知 | `UserMsg` | 把你的输入包装成 Agent 能理解的消息 |
| ② 思考 | `DeepSeekChatModel` | 大模型负责理解和生成 |
| ③ 决策 | `ReActConfig` | Agent 的"思考框架"，决定思考方式 |
| ③ 执行 | `FunctionTool` | 把普通函数变成 Agent 可用的工具 |
| ③ 权限 | `_confirm_tool_calls` | 写操作前需要用户确认 |
| ④ 输出 | `StreamConsoleRenderer` | 流式渲染 Agent 的回答和工具调用 |

### 7.3 AgentScope 还能做什么？

我们的项目基于 AgentScope 框架。AgentScope 是一个完整的 Agent 开发框架，除了我们用到的基础功能外，还有更多能力：

**多 Agent 协作**
```
多个 Agent 可以分工合作：
- 一个 Agent 负责读取文档
- 一个 Agent 负责生成内容
- 一个 Agent 负责审核质量
- 一个 Agent 负责最终输出

就像一支团队，各司其职。
```

**记忆系统**
```
Agent 可以记住之前的对话和操作：
- 短期记忆：当前对话的内容
- 长期记忆：跨对话的知识和偏好
- 文件记忆：从文件中读取的信息
```

**外部工具集成**
```
Agent 可以接入各种外部服务：
- 搜索引擎（查资料）
- 数据库（查数据）
- API（调服务）
- 代码执行器（运行代码）
```

### 7.4 从小龙虾看未来：Agent 的可能性

还记得我们在第 3 章聊的小龙虾（Manus）吗？它火爆全网不是偶然。它让人们第一次直观感受到：一个 AI 能像人类一样"干活"——不是回答问题，而是完成任务。

小龙虾和我们的 Agent 对比：

```
             小龙虾（Manus）             我们的 Markdown Agent
工具数量：    几十个                      9 个
工具类型：    浏览器、代码执行器、文件操作等    扫描、读取、编辑、格式化
能力范围：    研究、编程、数据分析、报告撰写     Markdown 文件处理
核心原理：    LLM + 工具 + 自主决策           LLM + 工具 + 自主决策
```

**核心原理完全相同。**

这意味着什么？意味着你理解了我们的 Agent，就理解了小龙虾这类产品背后的底层逻辑。区别只是工具多少、场景不同。如果你想让自己的 Agent 也有小龙虾的能力，只需要不断给它加工具：

```
我们的 Agent（9 个工具）
  → 加上浏览器工具 → 能上网搜索
  → 加上 Python 执行器 → 能运行代码
  → 加上数据库工具 → 能查询数据
  → 加上文件转换工具 → 能处理 PDF/Excel
  → 加上 API 工具 → 能调用外部服务
  → ……你想要的任何能力
```

**这就是从 0 到 1 搭建 Agent 的真正价值**：你不再是一个"使用者"，而是一个"建造者"。你可以打造属于自己的 Agent。
