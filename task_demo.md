我建议这个新课场景就围绕你现有的 `Test.md` 做：

**场景名称：把需求文档整理成“可验收版本”**

这个场景特别适合讲 Task tools，因为它不是单步任务，也不是简单“读一下文件”。它有多个目标、多个工具、多个写操作，还很容易让笨一点的模型跑偏。

你给 Agent 的用户需求可以设计成这样：

```text
查看记忆中我当前打开的 Markdown 文件。你现在需要把它整理成一个可验收版本。

请使用 Plan Mode 解决问题，不要一上来直接修改文件。

具体要求如下：

1. 先读取并理解文档整体结构，输出你识别到的主要章节结构和修改计划。
2. 不要修改以下受保护章节的任何正文内容：

   * `## 1. 背景`
   * `## 2. 用户场景`
   * `## 7. FAQ`
3. 精确读取以下章节，确认它们的当前位置和内容范围：

   * `### 3.3 插入新章节`
   * `## 4. 数据来源`
   * `### 4.1 来源校验`
   * `## 5. 风险`
   * `## 6. 待办清单`
   * `## 8. 验收标准`
4. 重写 `## 5. 风险` 章节，但保留 `## 5. 风险` 这个二级标题。

   * 重写后的风险章节至少包含 5 条风险。
   * 每条风险都必须同时包含：

     * `风险：`
     * `缓解方式：`
   * 风险描述要比原文更清晰，不能只列现象。
   * 可以删除该章节内部原有的 `### 目标` 小标题。
   * 不要误修改 `### 1.2 目标`，也不要修改 FAQ 中的内容。
5. 在 `### 3.3 插入新章节` 的正文之后、`### 3.4 更新代码块` 之前插入一个新小节：

   * 标题使用：`#### 本次修改要点`
   * 内容需要概括本次修改涉及的三个动作：

     * 重写风险章节
     * 插入数据更新频率说明
     * 同步更新待办清单
6. 在 `## 4. 数据来源` 章节内插入一个新小节：

   * 标题使用：`### 4.2 数据更新频率`
   * 插入位置必须在 `### 4.1 来源校验` 之后、`## 5. 风险` 之前。
   * 内容需要用表格说明不同数据来源的建议更新频率。
   * 表格至少包含这几列：

     * 数据来源
     * 建议更新频率
     * 更新触发条件
     * 负责人
7. 更新 `## 6. 待办清单` 中与本次修改相关的任务状态：

   * 将“支持替换风险章节”标记为已完成。
   * 将“支持在数据来源章节中插入更新频率说明”标记为已完成。
   * 将“支持在 3.3 后插入本次修改要点”标记为已完成。
   * 不要修改“支持更新 Python 示例代码”的状态。
   * 不要修改嵌套任务的状态。
8. 识别代码块中的伪标题。

   * 文档代码块里有字符串 `## 5. 风险`。
   * 这只是 Python 字符串内容，不是真实 Markdown 标题。
   * 不要把它当作章节标题。
   * 不要修改代码块内容。
9. 最后检查 Markdown 格式。

   * 统一标题前后空行。
   * 规范表格格式。
   * 规范任务列表缩进。
   * 不要因为格式化而改动受保护章节的正文含义。
10. 修改完成后，输出验收报告，至少包括：

* 你读取了哪些章节。
* 你替换了哪个章节。
* 你插入了哪些新小节，分别插入到哪里。
* 你更新了哪些任务状态。
* 你确认没有修改哪些受保护章节。
* 你是否发现并正确忽略了代码块中的伪标题。

注意事项：

* 不要全文重写。
* 不要使用全局字符串替换。
* 必须基于 heading path 或 section 范围进行精确编辑。
* 如果遇到重名标题，必须使用完整标题路径判断。
* 如果某一步无法完成，需要明确说明原因，而不是猜测修改。

```

**为什么这个场景好**

它天然需要 Task 管理。

如果不用 Task，笨一点的模型很容易出现这些问题：

- 一上来就直接改文件，没有先读结构。
- 改完“风险”章节后忘记插入“数据更新频率”。
- 插入了新内容，但忘记更新待办清单。
- 更新了一个任务，忘记另外两个任务。
- 写操作确认之后，模型丢失原来的目标。
- 最后直接说“完成了”，但没有检查格式。
- 看到多个“目标”标题时，有概率跑去错误章节。

而 Task tools 可以把这个任务变成一个稳定流程：

```text
TaskCreate:
1. 扫描并理解 Test.md 的文档结构
2. 读取“5. 风险”章节
3. 改写“5. 风险”章节
4. 在“4. 数据来源”后插入“数据更新频率”
5. 更新相关待办任务状态
6. 检查并格式化 Markdown
7. 汇总修改结果
```

这样你就能讲清楚：

> Task 工具不是替 Agent 做事，而是防止 Agent 忘事、跳步、跑偏。

**课堂演示方式**

第一轮可以故意不用 Task tools，只让小模型直接执行这个需求。

你让学生观察：

```text
它有没有先看结构？
它有没有完整执行 6 个要求？
它有没有忘记最后检查？
它有没有在写操作后丢目标？
```

第二轮接入 `TaskCreate / TaskList / TaskGet / TaskUpdate`。

让 Agent 先创建任务列表，再每完成一步就 `TaskUpdate`：

```text
Task 1: completed - 已读取文档结构
Task 2: completed - 已读取风险章节
Task 3: completed - 已替换风险章节
Task 4: completed - 已插入数据更新频率
Task 5: completed - 已更新 3 个待办项
Task 6: completed - 已检查/格式化
Task 7: completed - 已汇总结果
```

对学生来说，差异会非常直观：

```text
不用 Task：Agent 像是在边走边想，容易忘。
用了 Task：Agent 像有一张施工清单，做一步勾一步。
```

**这个场景最核心的教学点**

你可以把这一课的核心结论定成：

> ReAct 解决“下一步调用什么工具”，Task 解决“整个任务还剩哪些步骤”。

这句话很有力。

也可以画成：

```text
用户复杂目标
  ↓
TaskCreate：拆成任务清单
  ↓
ReAct：针对当前任务选择工具
  ↓
Markdown 工具：实际读写文件
  ↓
TaskUpdate：记录完成状态
  ↓
TaskList：检查是否还有遗漏
  ↓
最终汇报
```

**我最推荐的讲课标题**

```text
第 X 章：让 Agent 不跑偏 —— 用 Task 工具管理复杂任务
```

或者：

```text
第 X 章：从会调用工具，到会完成任务
```

这个场景和你当前项目贴得很紧，不需要引入 RAG、多 Agent、浏览器这些新变量，但能明显注入一个新知识点：**Planning as tools on top of ReAct**。
