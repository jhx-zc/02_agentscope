我建议这个新课场景就围绕你现有的 `Test.md` 做：

**场景名称：把需求文档整理成“可验收版本”**

这个场景特别适合讲 Task tools，因为它不是单步任务，也不是简单“读一下文件”。它有多个目标、多个工具、多个写操作，还很容易让笨一点的模型跑偏。

你给 Agent 的用户需求可以设计成这样：

```text
查看记忆中我当前打开的 Markdown 文件。你现在需要把它整理成一个基础可验收版本。

请使用 Plan Mode 解决问题，不要一开始就直接修改文件。

具体要求如下：

1. 先读取并理解文档整体结构，简单说明你识别到的主要章节。
2. 不要修改以下章节的正文内容：

   * `## 1. 背景`
   * `## 7. FAQ`
3. 读取并确认以下章节内容：

   * `### 3.3 插入新章节`
   * `## 4. 数据来源`
   * `## 5. 风险`
   * `## 6. 待办清单`
4. 重写 `## 5. 风险` 章节，但保留 `## 5. 风险` 这个标题。

   * 重写后保留 3 条风险。
   * 每条风险都要包含：

     * `风险：`
     * `缓解方式：`
   * 风险描述要比原文更清楚。
5. 在 `### 3.3 插入新章节` 后插入一个新小节：

   * 标题：`#### 本次修改要点`
   * 内容简要说明本次修改包括：重写风险章节、插入数据更新频率说明、更新待办清单。
6. 在 `## 4. 数据来源` 后插入一个新小节：

   * 标题：`### 4.1 数据更新频率`
   * 用一个简单表格说明不同数据来源的更新频率。
   * 表格包含三列：

     * 数据来源
     * 更新频率
     * 说明
7. 更新 `## 6. 待办清单` 中的任务状态：

   * 将“支持替换风险章节”标记为已完成。
   * 将“支持插入数据更新频率说明”标记为已完成。
   * 其他任务状态保持不变。
8. 最后检查 Markdown 格式：

   * 标题前后保留空行。
   * 表格格式保持规范。
   * 任务列表格式保持规范。
9. 修改完成后，告诉我：

   * 读取了哪些章节。
   * 修改了哪个章节。
   * 插入了哪些新小节。
   * 更新了哪些任务状态。
   * 确认没有修改哪些受保护章节。

注意：

* 不要全文重写。
* 不要修改背景和 FAQ。
* 不要使用全局字符串替换。
* 尽量基于标题路径或 section 范围进行修改。


```

**为什么这个场景好**

它天然需要 Task 管理。

如果不用 Task，笨一点的模型很容易出现这些问题：

- 一上来就直接改文件，没有先读结构。
- 改完“风险”章节后忘记插入“数据更新频率”。
- 插入了新内容，但忘记更新待办清单。
- 更新了一个任务，忘记另外两个任务。
- 写操作确认之后，模型丢失原来的目标。
- 最后直接说“完成了”，但没有检查格式。
- 看到多个“目标”标题时，有概率跑去错误章节。

而 Task tools 可以把这个任务变成一个稳定流程：

```text
TaskCreate:
1. 扫描并理解 Test.md 的文档结构
2. 读取“5. 风险”章节
3. 改写“5. 风险”章节
4. 在“4. 数据来源”后插入“数据更新频率”
5. 更新相关待办任务状态
6. 检查并格式化 Markdown
7. 汇总修改结果
```

这样你就能讲清楚：

> Task 工具不是替 Agent 做事，而是防止 Agent 忘事、跳步、跑偏。

**课堂演示方式**

第一轮可以故意不用 Task tools，只让小模型直接执行这个需求。

你让学生观察：

```text
它有没有先看结构？
它有没有完整执行 6 个要求？
它有没有忘记最后检查？
它有没有在写操作后丢目标？
```

第二轮接入 `TaskCreate / TaskList / TaskGet / TaskUpdate`。

让 Agent 先创建任务列表，再每完成一步就 `TaskUpdate`：

```text
Task 1: completed - 已读取文档结构
Task 2: completed - 已读取风险章节
Task 3: completed - 已替换风险章节
Task 4: completed - 已插入数据更新频率
Task 5: completed - 已更新 3 个待办项
Task 6: completed - 已检查/格式化
Task 7: completed - 已汇总结果
```

对学生来说，差异会非常直观：

```text
不用 Task：Agent 像是在边走边想，容易忘。
用了 Task：Agent 像有一张施工清单，做一步勾一步。
```

**这个场景最核心的教学点**

你可以把这一课的核心结论定成：

> ReAct 解决“下一步调用什么工具”，Task 解决“整个任务还剩哪些步骤”。

这句话很有力。

也可以画成：

```text
用户复杂目标
  ↓
TaskCreate：拆成任务清单
  ↓
ReAct：针对当前任务选择工具
  ↓
Markdown 工具：实际读写文件
  ↓
TaskUpdate：记录完成状态
  ↓
TaskList：检查是否还有遗漏
  ↓
最终汇报
```

**我最推荐的讲课标题**

```text
第 X 章：让 Agent 不跑偏 —— 用 Task 工具管理复杂任务
```

或者：

```text
第 X 章：从会调用工具，到会完成任务
```

这个场景和你当前项目贴得很紧，不需要引入 RAG、多 Agent、浏览器这些新变量，但能明显注入一个新知识点：**Planning as tools on top of ReAct**。
